SECTION 2: CI/CD Pipelines ▼

What it is: It automates building, testing, and deploying code every time a developer pushes changes.
CI (Continuous Integration): Automatically builds and tests code on every push/PR to catch bugs early.
CD (Continuous Delivery): Automatically deploys the tested artifact to one or more environments (Dev → Staging → Production).
Flow: Developer pushes code → Azure Repo → CI Pipeline triggers → Build + Test → Publish Artifact → CD Pipeline → Deploy to Azure App Service
Why needed: Eliminates manual deployments, catches bugs early, ensures consistent releases, enables zero-downtime deployments with slot swapping.

Component Description Azure DevOps Term
CI Pipeline Triggered on code push or PR. Compiles code, runs unit tests, and publishes a deployable artifact (zip file). Pipelines → Pipelines (YAML or Classic)
CD Pipeline Picks up the artifact from CI and deploys it to target environments. Can have approval gates before production. Pipelines → Releases (Classic) or multi-stage YAML
Artifact The output of the CI pipeline — a published, compiled, ready-to-deploy zip of your application. Published via DotNetCoreCLI@2 publish task
Service Connection Authenticates Azure DevOps to deploy into your Azure Subscription. Must be created before pipeline can deploy. Project Settings → Service Connections → Azure Resource Manager
Agent Pool The virtual machine that runs your pipeline tasks. Microsoft-hosted (ubuntu-latest, windows-latest) or self-hosted. pool: vmImage: 'ubuntu-latest'

Azure Repos – Setup & Push Code:

  1. Go to dev.azure.com → Select your Organization & Project → Left sidebar → Repos
  2. Click dropdown at top → New Repository → Type: Git → Enter name → Click Create
  3. Clone repo locally: git clone https://dev.azure.com/{org}/{project}/_git/{repo}
  4. Or add remote to existing project: git remote add origin https://dev.azure.com/{org}/{project}/_git/{repo}
  5. Push code: git add . → git commit -m "message" → git push -u origin main
  6. For authentication use PAT (Personal Access Token) → User Settings → Personal Access Tokens → New Token → Scopes: Code (Read & Write)
Branch Policy Description
Require Pull Request No direct push to main/master allowed. All changes must go through a Pull Request.
Minimum Reviewers PR requires at least N approvals before it can be merged.
Build Validation CI pipeline must pass successfully before the PR can be merged into the protected branch.
Comment Resolution All PR review comments must be resolved before merge is allowed.
Linked Work Items PR must be linked to a task or user story from Azure Boards.

CI Pipeline – Steps to Create (Classic / YAML):

  1. Azure DevOps → Pipelines → New Pipeline
  2. Select source: Azure Repos Git → choose your repository
  3. Choose template: ASP.NET Core or start with empty YAML
  4. Add DotNetCoreCLI@2 task → command: restore → restores NuGet packages
  5. Add DotNetCoreCLI@2 task → command: build → compiles the project
  6. Add DotNetCoreCLI@2 task → command: test → runs unit tests (optional but recommended)
  7. Add DotNetCoreCLI@2 task → command: publish → creates deployable artifact zip
  8. Add Publish Pipeline Artifact task → publishes the zip so CD can pick it up
  9. Set trigger: trigger: - main → pipeline auto-runs on every push to main branch
  10. Save and Run → verify build succeeds in pipeline logs

CI Pipeline – YAML Example (azure-pipelines.yml): 

trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

variables:
  buildConfiguration: 'Release'

steps:

  - task: DotNetCoreCLI@2
    displayName: 'Restore'
    inputs:
      command: 'restore'
      projects: '**/*.csproj'

  - task: DotNetCoreCLI@2
    displayName: 'Build'
    inputs:
      command: 'build'
      projects: '**/*.csproj'
      arguments: '--configuration $(buildConfiguration)'

  - task: DotNetCoreCLI@2
    displayName: 'Test'
    inputs:
      command: 'test'
      projects: '**/*Tests.csproj'

  - task: DotNetCoreCLI@2
    displayName: 'Publish'
    inputs:
      command: 'publish'
      publishWebProjects: true
      arguments: '--configuration $(buildConfiguration) --output $(Build.ArtifactStagingDirectory)'
      zipAfterPublish: true

  - task: PublishPipelineArtifact@1
    displayName: 'Publish Artifact'
    inputs:
      targetPath: '$(Build.ArtifactStagingDirectory)'
      artifact: 'drop'

CD Pipeline – Steps to Create (Release Pipeline):

  1. Azure DevOps → Pipelines → Releases → New Pipeline
  2. Click Add an artifact → Source: Build → Select your CI pipeline → Default version: Latest → Click Add
  3. Enable Continuous Deployment Trigger (lightning bolt icon on artifact) → so CD triggers automatically after CI succeeds
  4. Click Add a Stage → Choose template: Azure App Service Deployment
  5. In stage tasks → Set Azure Subscription → select your Service Connection
  6. Set App Service Name → select your App Service from the dropdown
  7. Set Deployment Slot (optional) → e.g. staging for slot-based deployments
  8. Package: $(System.DefaultWorkingDirectory)/**/*.zip
  9. For Production: add a second stage → add Pre-deployment Approval → assign approver email
  10. After staging deploy is validated → add Azure App Service Manage task → Action: Swap Slots → swap staging with production
  11. Save → Create Release → Monitor deployment in Releases tab

CD – Multi-Stage YAML Example (deploy stage added to same file): 

stages:
  - stage: Build
    jobs:
      - job: BuildJob
        steps:
          # ... (same as CI steps above)

  - stage: Deploy_Staging
    dependsOn: Build
    condition: succeeded()
    jobs:
      - deployment: DeployToStaging
        environment: 'Staging'
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureWebApp@1
                  displayName: 'Deploy to Staging Slot'
                  inputs:
                    azureSubscription: 'MyServiceConnection'
                    appType: 'webApp'
                    appName: 'my-app-service'
                    deployToSlotOrASE: true
                    resourceGroupName: 'my-rg'
                    slotName: 'staging'
                    package: '$(Pipeline.Workspace)/drop/**/*.zip'

  - stage: Deploy_Production
    dependsOn: Deploy_Staging
    condition: succeeded()
    jobs:
      - deployment: SwapToProduction
        environment: 'Production'   # environment has approval gate configured
        strategy:
          runOnce:
            deploy:
              steps:
                - task: AzureAppServiceManage@0
                  displayName: 'Swap Staging to Production'
                  inputs:
                    azureSubscription: 'MyServiceConnection'
                    Action: 'Swap Slots'
                    WebAppName: 'my-app-service'
                    ResourceGroupName: 'my-rg'
                    SourceSlot: 'staging'
Important Setting Description
trigger Defines which branch push auto-triggers the CI pipeline. Use trigger: - main or specify multiple branches.
pr (PR trigger) Runs pipeline when a Pull Request is raised. Use pr: - main to validate PRs before merge. Works with Build Validation branch policy.
variables Store reusable values like buildConfiguration: Release. Reference with $(buildConfiguration). Sensitive values go in Variable Groups (Library) — never hardcode secrets.
Service Connection Must be created before deployment tasks. Project Settings → Service Connections → New → Azure Resource Manager → choose Subscription → grant access.
Environment + Approvals In Pipelines → Environments → create "Production" environment → Add Approval → set approver. When YAML deploy stage targets this environment, it pauses and waits for manual approval before deploying.
Variable Groups (Library) Pipelines → Library → Variable Groups → store shared secrets (DB passwords, API keys) linked across multiple pipelines. Can link to Azure Key Vault to pull secrets directly.
Deployment Slot (Swap) Deploy to staging slot first → test → use AzureAppServiceManage@0 Swap task to promote to production with zero downtime. Staging slot gets production traffic instantly after swap.
condition Controls when a stage/step runs. condition: succeeded() only runs if previous stage passed. condition: always() runs even on failure (useful for cleanup steps).
dependsOn Defines stage execution order in multi-stage YAML. dependsOn: Build means Deploy stage only starts after Build stage completes successfully.
zipAfterPublish In DotNetCoreCLI@2 publish task — set to true to zip the output. The AzureWebApp@1 deploy task expects a zip file.

How to Deploy to Azure App Service (Full Flow):

  1. Azure Portal → Create App Service → Choose Runtime: .NET 8 → Select App Service Plan → Click Create
  2. In App Service → Deployment Center → Source: Azure DevOps → Select Org, Project, Repo, Branch → Save (this can auto-generate a pipeline)
  3. Or manually: in your YAML pipeline add AzureWebApp@1 task with your Service Connection and App Name
  4. In App Service → Configuration → Application Settings → Add ASPNETCORE_ENVIRONMENT, ConnectionStrings, API keys (overrides appsettings.json)
  5. Create a Staging slot: App Service → Deployment Slots → Add Slot → Name: staging → Clone settings from production
  6. CD Pipeline deploys artifact to staging slot first → run smoke tests / manual validation
  7. Then run AzureAppServiceManage@0 Swap task → swaps staging ↔ production → zero downtime
  8. Monitor after deploy: App Service → Application Insights → check for errors, slow requests, exceptions

Dockerfile — Line by Line:

FROM mcr.microsoft.com/dotnet/sdk:7.0 AS build
# Start with .NET SDK image as base (has the compiler)
# "AS build" = name this stage "build" for multi-stage use

WORKDIR /src
# Set working directory inside container to /src

COPY . .
# Copy all files from your computer into /src in container

RUN dotnet publish -c Release -o /app
# Compile in Release mode; output goes to /app

FROM mcr.microsoft.com/dotnet/aspnet:7.0
# New stage — lightweight runtime only (no compiler)
# Final image is much smaller than SDK image

WORKDIR /app

COPY --from=build /app .
# Copy only compiled output from the "build" stage

EXPOSE 80
# Container listens on port 80

ENTRYPOINT ["dotnet", "MyApp.dll"]
# Run this when container starts
Without Multi-Stage Build With Multi-Stage Build
Final image includes SDK (~1 GB) Final image has runtime only (~200 MB) — 5× smaller
Includes compiler and build tools not needed at runtime Only what's needed to run the app
Slower deployment, more storage cost Faster deployment, cheaper cloud storage

Docker Layer Caching:

  • Each Dockerfile line = one layer (filesystem snapshot)
  • Unchanged layers are reused from cache — builds in seconds instead of minutes
  • Put frequently changing commands LAST — copy source code after installing dependencies, so the install layer is cached and reused even when code changes
  • Smaller image = faster everything — less download time, faster deployments, lower storage cost

SECTION 4: Kubernetes (K8s) ▼

What is Kubernetes? Automatic manager for Docker containers. Handles running, scaling, updating, and healing containers across multiple machines. You declare what you want — K8s figures out how to make it happen.

  • Use K8s when: Many containers running across multiple servers at scale (like Netflix, Spotify)
  • Don't use K8s when: 1–2 containers on a single machine — just use Docker or App Service
Component What it does Real-world example
Cluster Group of machines (servers) running Kubernetes together 10 servers in a data center = 1 K8s cluster
Node Single machine/server inside the cluster that runs containers One physical or virtual server
Pod Smallest K8s unit — 1+ containers running together Usually 1 container per pod (your API)
Deployment Instructions: "run 5 copies of my API, keep them running always" You declare 5 replicas → K8s creates and manages them
Service Load balancer in front of multiple pods — single endpoint, balanced traffic 5 API pods behind one URL — traffic split between them
ReplicaSet Ensures desired number of pods are always running 1 pod crashes → ReplicaSet creates a replacement immediately
ConfigMap Store non-secret config (database URL, environment name) Dev vs Prod can have different DB URLs without rebuilding image
Secret Store sensitive config (passwords, API keys) encrypted DB password injected at runtime — never in code
Ingress External HTTP entry point — routes traffic to the right service api.example.com → routes to API service
Namespace Logical isolation inside cluster — separate dev, staging, prod Prevent accidents, easier management per environment

How Kubernetes Works — Step-by-Step:

  1. Write a Deployment YAML — "run 5 replicas of myapi:1.0, listen on port 80"
  2. Apply it: kubectl apply -f deployment.yaml
  3. K8s schedules pods across available nodes, distributing load evenly
  4. Each pod pulls the Docker image from ACR and starts the container
  5. A Service sits in front of all 5 pods — single entry point, load balanced
  6. K8s continuously health-checks pods — if one crashes, a new one is created automatically
  7. Scale up: kubectl scale deployment myapi --replicas=10 → 5 more pods instantly
  8. Rolling update: new image available → K8s replaces pods one at a time → zero downtime

Basic Deployment YAML:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapi
spec:
  replicas: 5                        # Run 5 copies
  selector:
    matchLabels:
      app: myapi                     # Manage pods with this label
  template:
    metadata:
      labels:
        app: myapi
    spec:
      containers:
      - name: myapi
        image: acr.azurecr.io/myapi:1.0   # Always use specific tag, never "latest"
        ports:
        - containerPort: 80
        env:
        - name: DATABASE_URL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: db-url            # Config from ConfigMap, not hardcoded

Tool What it is When to use Complexity
Docker Package app into container Dev, testing, single server deployment ⭐ Easy
Azure App Service Managed hosting — Azure handles everything Small-medium app, no server management ⭐⭐ Medium
Kubernetes (AKS) Orchestrate 100s–1000s of containers across many servers Large scale, auto-scaling, complex deployments ⭐⭐⭐ Hard

Kubernetes Best Practices:

  • Never use latest tag — always specify version (myapi:1.2.3). latest is unpredictable across different servers.
  • Set resource limits — specify CPU and memory per pod. Prevents one pod from consuming all resources and crashing others.
  • Add liveness probes — K8s pings your health endpoint. Unhealthy pods are restarted automatically.
  • Use Namespaces — separate dev, staging, prod to prevent accidents and ease management.
  • Use Secrets for credentials — passwords and API keys are encrypted; never put them in YAML files.
  • Use ConfigMaps for config — change settings without rebuilding the image.

SECTION 5: Complete DevOps Workflow (End-to-End) ▼

Phase What happens
DEVELOP Write C# API → create Dockerfile → write pipeline YAML → git push origin main
BUILD Git push triggers pipeline → dotnet build → dotnet test → build Docker image → push to ACR with version tag
DEPLOY Auto-deploy to Dev → Approval gate → Deploy to Staging → QA testing → Final approval → Deploy to Production
RUN Kubernetes / App Service runs container → 5 replicas load-balanced → K8s health checks → auto-heal on crash → auto-scale on traffic spike → Application Insights monitors everything

Full Flow in One Line:

Developer pushes code → Pipeline builds & tests → Docker image pushed to ACR → Deployed to Dev → Approval → Staging → Approval → Production → Kubernetes runs, heals, and scales automatically → Application Insights monitors
