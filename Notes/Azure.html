1. Frontend Hosting ‚Äì Azure Static Web Apps
‚ñº
What it is:

A managed service to host your Angular, React, or Vue web applications. It automatically provides HTTPS (secure connection), fast global delivery using CDN, and easy automatic deployment when you push code to GitHub.

Why use it:
  • ‚úì Automatic HTTPS (secure by default)
  • ‚úì Global CDN for fast loading worldwide
  • ‚úì Auto-deploy when you push code to GitHub
  • ‚úì Built-in security and backups
  • ‚úì No server management needed
How to implement (Step by step) ‚ñº
  1. Step 1: Build your Angular project on your computer ‚Üí creates dist/ folder
  2. Step 2: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Static Web App" ‚Üí Click Create
  3. Step 3: Sign in to GitHub and authorize Azure to access your repository
  4. Step 4: Select your GitHub repository and branch (usually "main")
  5. Step 5: Set App location: / and Output location: dist/project-name (example: dist/my-angular-app)
  6. Step 6: Click "Review + Create" ‚Üí Click "Create"
  7. Step 7: Azure automatically deploys whenever you push code to GitHub
  8. Step 8: In your Angular app, update environment.prod.ts with your API base URL (example: https://your-api.azurewebsites.net)
  9. Step 9: In your backend API, enable CORS to allow requests from your Static Web App domain
User Browser ‚Üí Static Web App (Fast CDN) ‚Üí Backend API

2. Backend Hosting ‚Äì Azure App Service
‚ñº
What it is:

A fully managed platform to host your .NET, Node.js, Java, or Python web applications and APIs. Azure automatically handles server maintenance, updates, and scales up or down based on traffic demand.

Why use it:
  • ‚úì No server management ‚Äì Azure handles everything
  • ‚úì Automatic scaling ‚Äì handles traffic spikes automatically
  • ‚úì Easy deployment from Visual Studio or Azure DevOps
  • ‚úì Built-in monitoring and health checks
  • ‚úì Supports multiple languages (.NET, Node.js, Java, Python)
  • ‚úì Multiple APIs can share same resource to save costs
What is App Service Plan?

An App Service Plan defines your compute resources (CPU, RAM, pricing tier, and auto-scaling rules). Think of it as the "power level" of your server. Multiple APIs can share the same App Service Plan to save costs without affecting each other.

How to implement (Step by step) ‚ñº
  1. Step 1: Publish your .NET API in Visual Studio ‚Üí right-click project ‚Üí Publish ‚Üí creates publish folder
  2. Step 2: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "App Service" ‚Üí Click Create
  3. Step 3: Fill in details: Resource Group, App name (example: my-api), Runtime stack (.NET 6.0)
  4. Step 4: Select App Service Plan or create new one with your preferred pricing tier
  5. Step 5: Click "Review + Create" ‚Üí Click "Create"
  6. Step 6: After deployment, go to App Service ‚Üí Deployment Center ‚Üí Connect GitHub or Azure DevOps
  7. Step 7: Configure auto-deployment so code pushes automatically deploy
  8. Step 8: Go to App Service ‚Üí SSL Settings ‚Üí Enable "HTTPS Only" for security
  9. Step 9: Optional: Configure auto-scaling rules based on CPU/memory usage
Configuration Priority (Important!)
1. Environment Variables (highest priority ‚Äì overrides everything)
2. App Service Configuration (in Azure Portal settings)
3. appsettings.json (lowest priority)

3. Database ‚Äì Azure SQL Database
‚ñº
What it is:

A managed cloud SQL Server database that automatically backs up your data every day, provides high availability (keeps running even if hardware fails), and applies security patches automatically.

Why use it:
  • ‚úì Automatic daily backups (never lose data)
  • ‚úì High availability and disaster recovery (always available)
  • ‚úì Automatic security patches (no manual updates needed)
  • ‚úì Easy scaling (increase storage/performance when needed)
  • ‚úì No database maintenance overhead (Azure handles everything)
How to implement (Step by step) ‚ñº
  1. Step 1: Export your local database as .bacpac file (right-click database in SSMS ‚Üí Tasks ‚Üí Export)
  2. Step 2: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "SQL Database" ‚Üí Click Create
  3. Step 3: Fill in details: Resource Group, Database name, Server (create new or select existing)
  4. Step 4: Choose pricing tier based on your needs (Standard, Premium, etc.)
  5. Step 5: Click "Review + Create" ‚Üí Click "Create"
  6. Step 6: After creation, open SQL Server Management Studio (SSMS) on your computer
  7. Step 7: In SSMS, right-click database ‚Üí Tasks ‚Üí Import Data-tier Application
  8. Step 8: Select your .bacpac file and follow the wizard to import your data
  9. Step 9: Go to Azure Portal ‚Üí SQL Database ‚Üí Firewall settings ‚Üí Allow your IP address to connect

4. Secrets Management ‚Äì Azure Key Vault + Managed Identity
‚ñº
What it is:

A secure vault to store secrets like database passwords, API keys, and connection strings. Never hardcode secrets in your code! Managed Identity allows your App Service to securely access the Key Vault without storing passwords.

Why use it:
  • ‚úì No secrets in your code (safe and secure)
  • ‚úì Centralized secret management (one place to manage all secrets)
  • ‚úì Secure access control (only authorized services can access)
  • ‚úì Audit logs (see who accessed what and when)
  • ‚úì Easy secret rotation (update secrets without redeploying)
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Key Vault" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Vault name (example: my-keyvault)
  3. Step 3: Click "Review + Create" ‚Üí Click "Create"
  4. Step 4: Open Key Vault ‚Üí Click "Secrets" ‚Üí Click "Generate/Import"
  5. Step 5: Add your secrets (database connection string, API keys, etc.)
  6. Step 6: Go to your App Service ‚Üí Identity ‚Üí Enable "System Assigned" Managed Identity
  7. Step 7: Copy the Managed Identity Object ID
  8. Step 8: Go to Key Vault ‚Üí Access Policies ‚Üí Click "Add Access Policy"
  9. Step 9: Select Secret permissions (Get, List) ‚Üí Select Principal (paste Object ID) ‚Üí Click Add
  10. Step 10: In App Service ‚Üí Configuration ‚Üí Add Key Vault references like @Microsoft.KeyVault(VaultName=my-keyvault;SecretName=DbPassword)
App Service (Managed Identity enabled) ‚Üí Key Vault access role ‚Üí Reads secrets securely

5. API Gateway ‚Äì Azure API Management (APIM)
‚ñº
What it is:

A gateway service that sits in front of all your microservices. It provides centralized security, rate limiting (prevents spam/abuse), JWT token validation, and API versioning. All API traffic goes through APIM first.

Why use it:
  • ‚úì Centralized security policies (manage security in one place)
  • ‚úì Rate limiting (prevent spam and abuse)
  • ‚úì JWT token validation (verify user is authenticated)
  • ‚úì API versioning (support multiple API versions simultaneously)
  • ‚úì Comprehensive logging and monitoring (track all API calls)
  • ‚úì CORS handling (allow cross-origin requests safely)
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "API Management" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, API Management name (example: my-apim)
  3. Step 3: Select pricing tier (Developer is cheapest for testing)
  4. Step 4: Click "Review + Create" ‚Üí Click "Create" (takes 20-30 minutes)
  5. Step 5: After creation, go to APIM ‚Üí APIs ‚Üí Click "Add API"
  6. Step 6: Select "Manual" ‚Üí Enter API name (example: ProductApi) and Web service URL (your App Service URL)
  7. Step 7: Click "Create" to add the API
  8. Step 8: Go to Inbound policies ‚Üí Add "JWT Validate" policy to check tokens
  9. Step 9: Add "Rate Limit" policy to prevent abuse (example: 100 calls per 1 minute)
  10. Step 10: In your Angular app, change API URL from your App Service to APIM endpoint (example: https://my-apim.azure-api.net/productapi)
Angular UI ‚Üí APIM Gateway ‚Üí Microservices (App Service, Functions)

6. File Storage ‚Äì Azure Blob Storage
‚ñº
What it is:

Object storage service for storing files (images, PDFs, videos, documents) at scale. Don't store files in your database ‚Äì store them in Blob Storage which is cheaper and faster. Each file has a unique URL for download.

Why use it:
  • ‚úì Store and serve files at massive scale
  • ‚úì Separate file storage from database (database stays small and fast)
  • ‚úì Cost-effective for large files (much cheaper than database storage)
  • ‚úì Fast global delivery (files cached worldwide)
  • ‚úì Easy permission control (public, private, or time-limited access)
Container Access Levels:

Private: Requires authentication (recommended). Public Blob: Individual file accessible without login. Public Container: All files in container accessible.

How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Storage Account" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Storage account name (example: mystorageaccount)
  3. Step 3: Select replication (Locally-redundant is cheapest)
  4. Step 4: Click "Review + Create" ‚Üí Click "Create"
  5. Step 5: Go to Storage Account ‚Üí Containers ‚Üí Click "Add Container"
  6. Step 6: Enter container name (example: documents) ‚Üí Select "Private" access level ‚Üí Click Create
  7. Step 7: In your .NET API, install NuGet package: Azure.Storage.Blobs
  8. Step 8: Create API endpoint to upload files (receives file ‚Üí saves to Blob ‚Üí returns URL)
  9. Step 9: Create API endpoint to download files (generates SAS URL with time limit ‚Üí returns to user)
  10. Step 10: In Angular, call API to upload/download files using the returned URLs
Upload Flow:

Angular UI ‚Üí API Endpoint ‚Üí Upload to Blob Storage ‚Üí Save file metadata in SQL Database

Download Flow:

Angular UI ‚Üí Request file from API ‚Üí API generates SAS URL ‚Üí Angular downloads directly from Blob

What is SAS URL?

A temporary secure access token that allows limited-time access to files without exposing storage account keys. Example: You generate a SAS URL that expires in 1 hour, so user can download the file only for 1 hour.

7. Message Queues ‚Äì Azure Queue Storage
‚ñº
What it is:

A messaging service that lets different parts of your application communicate asynchronously. Instead of waiting for a task to complete, you add a message to a queue and an Azure Function processes it later. User gets instant response while background processing happens.

Why use it:
  • ‚úì Background processing (long tasks run in background)
  • ‚úì Decouple services (parts don't need to wait for each other)
  • ‚úì Avoid long API wait times (user gets instant response)
  • ‚úì Asynchronous task processing (tasks complete when ready)
  • ‚úì Automatic retry if processing fails (messages not lost)
How to implement (Step by step) ‚ñº
  1. Step 1: Go to Storage Account ‚Üí Queues ‚Üí Click "Add Queue"
  2. Step 2: Enter queue name (example: image-processing-queue) ‚Üí Click Create
  3. Step 3: In your .NET API, install NuGet package: Azure.Storage.Queues
  4. Step 4: Create API endpoint that receives file upload ‚Üí saves to Blob ‚Üí adds message to queue ‚Üí returns success to user immediately
  5. Step 5: Create Azure Function with "Queue Trigger" (listens for new messages in queue)
  6. Step 6: Azure Function receives message ‚Üí processes file ‚Üí updates database
  7. Step 7: If processing fails, message automatically retries (configurable max retries)
  8. Step 8: After max retries, message moves to Poison Queue for manual inspection
User ‚Üí API (Upload) ‚Üí Queue (Message Added) ‚Üí Azure Function (Processes) ‚Üí Database Update
Failure Handling:

Messages are automatically retried if processing fails (default: 5 times). After max retries, messages move to Poison Queue for inspection. You can manually retry or delete them.

8. Serverless Computing ‚Äì Azure Functions
‚ñº
What it is:

A serverless service that runs your code automatically when an event happens (message arrives, file uploaded, timer triggers, HTTP request, etc.). No server management needed ‚Äì Azure scales automatically and you pay only for the time your code runs (per millisecond).

Why use it:
  • ‚úì Event-driven processing (code runs when events happen)
  • ‚úì No server management (Azure handles everything)
  • ‚úì Automatic scaling (handles any load automatically)
  • ‚úì Pay only for execution time (no cost when idle)
  • ‚úì Perfect for background jobs and async tasks
Trigger Types (What starts the function):
  • ‚úì Queue Trigger ‚Äì triggered by message in Queue Storage
  • ‚úì Service Bus Trigger ‚Äì triggered by Service Bus message
  • ‚úì HTTP Trigger ‚Äì triggered by HTTP request (like API endpoint)
  • ‚úì Timer Trigger ‚Äì triggered on schedule (Cron expression like "0 0 * * * *")
  • ‚úì Blob Trigger ‚Äì triggered when file uploaded/modified in Blob Storage
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Function App" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Function App name (example: my-functions), Runtime (.NET 6.0)
  3. Step 3: Select hosting plan (Consumption plan is cheapest)
  4. Step 4: Click "Review + Create" ‚Üí Click "Create"
  5. Step 5: Open Function App ‚Üí Click "Add" or use Visual Studio to create new function
  6. Step 6: Select trigger type (example: Queue trigger) and connect to your queue
  7. Step 7: Write your function code (example: process image, send email, update database)
  8. Step 8: Test locally using Azure Functions Core Tools or deploy to Azure
  9. Step 9: For HTTP triggers, set authorization level (Anonymous = no key, Function = requires key)
  10. Step 10: Go to Function App ‚Üí Configuration ‚Üí Add connection strings for Blob Storage, SQL Database, etc.
HTTP Trigger Security:

Function Key: Acts like an API password. Pass via query string: ?code=FUNCTION_KEY or Header: x-functions-key: FUNCTION_KEY. Authorization Levels: Anonymous (no key), Function (function key required), Admin (master key).

9. Enterprise Messaging ‚Äì Azure Service Bus
‚ñº
What it is:

A reliable messaging service for different microservices or applications to communicate with guaranteed delivery. Unlike simple Queue Storage, Service Bus provides advanced features like guaranteed ordering, dead-letter queues, and transactions. Perfect for enterprise applications.

Why use it:
  • ‚úì Reliable communication between services (guaranteed delivery)
  • ‚úì Decouple microservices (parts don't need to know each other)
  • ‚úì Advanced messaging scenarios (pub/sub, transactions, ordering)
  • ‚úì Guaranteed delivery (messages never lost)
  • ‚úì Automatic retry and dead-letter handling
Types of Service Bus:
  • ‚úì Queue (1-to-1) ‚Äì one sender, one receiver gets each message
  • ‚úì Topic (1-to-many) ‚Äì one sender, multiple subscribers get same message
Features:
  • ‚úì Dead Letter Queue (DLQ) ‚Äì stores failed messages for inspection
  • ‚úì Message Ordering ‚Äì guarantees messages processed in order
  • ‚úì Automatic retry with backoff (waits longer between retries)
  • ‚úì Atomic transactions ‚Äì multiple operations succeed or all fail together
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Service Bus" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Namespace name (example: my-servicebus)
  3. Step 3: Select pricing tier (Basic is cheapest)
  4. Step 4: Click "Review + Create" ‚Üí Click "Create"
  5. Step 5: Go to Service Bus Namespace ‚Üí Click "Queues" ‚Üí Click "Add Queue"
  6. Step 6: Enter queue name and configure Dead Letter and Retry settings ‚Üí Click Create
  7. Step 7: In your .NET API, install NuGet package: Azure.Messaging.ServiceBus
  8. Step 8: Create sender code to send messages to Service Bus queue/topic
  9. Step 9: Create Azure Function with "Service Bus Trigger" to receive and process messages
  10. Step 10: Go to Service Bus Namespace ‚Üí Shared access policies ‚Üí Copy connection string for API and Function
Microservice 1 ‚Üí Service Bus Queue/Topic ‚Üí Microservice 2 & 3 (receive same message if Topic)

10. Performance Caching ‚Äì Azure Redis Cache
‚ñº
What it is:

A fast in-memory cache that stores frequently accessed data. Instead of querying database every time, you store data in Redis (memory), which is 1000x faster. Perfect for data that doesn't change often (product lists, user profiles, configuration).

Why use it:
  • ‚úì Reduce database load (database doesn't get hammered with same queries)
  • ‚úì Improve API response times (in-memory access is blazingly fast)
  • ‚úì Cache frequently accessed data (products, settings, user preferences)
  • ‚úì Session storage (remember user login across requests)
  • ‚úì Reduce costs (fewer database calls = lower database charges)
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Azure Cache for Redis" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Cache name (example: my-cache), Pricing tier (Basic is cheapest)
  3. Step 3: Click "Review + Create" ‚Üí Click "Create"
  4. Step 4: Go to Redis Cache ‚Üí Settings ‚Üí Access Keys ‚Üí Copy Primary Connection String
  5. Step 5: In your .NET API, install NuGet package: StackExchange.Redis
  6. Step 6: Create API endpoint with cache logic:
  7. Step 7: Check if data exists in Redis ‚Üí If yes, return immediately (Cache Hit - fast!)
  8. Step 8: If not, query database ‚Üí store result in Redis with TTL (Time To Live, example: 1 hour)
  9. Step 9: Return data to user
  10. Step 10: When data changes (update/delete), clear cache so next query fetches fresh data
Cache Hit: Angular calls API ‚Üí Found in Redis ‚Üí Return immediately (fast!)
Cache Miss: Not in Redis ‚Üí Query Database ‚Üí Save in Redis ‚Üí Return
Cache Strategy:

TTL (Time To Live): Set how long data stays in cache (example: 1 hour). After TTL expires, next request will fetch fresh data from database. This balances between performance (fast cached data) and freshness (not too stale).

11. User Authentication ‚Äì Azure Active Directory (AAD/Entra ID)
‚ñº
What it is:

Azure's identity service that manages user login and controls who can access what. Users login with username/password, Azure validates them, and gives your app a JWT token. Your API checks the token to verify user is authenticated before returning data.

Why use it:
  • ‚úì Secure user login system (don't store passwords yourself)
  • ‚úì Centralized identity management (one place to manage all users)
  • ‚úì Single Sign-On (SSO) (login once, access multiple apps)
  • ‚úì Role-based access control (Admin, User, Manager roles)
  • ‚úì Multi-Factor Authentication (MFA) (extra security with 2FA)
  • ‚úì Works with Office 365 accounts (users can login with their work email)
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Azure Active Directory" ‚Üí Click "App registrations" ‚Üí Click "New registration"
  2. Step 2: Enter app name (example: My-API) ‚Üí Click "Register"
  3. Step 3: Copy and save Tenant ID and Client ID
  4. Step 4: Go to "Expose an API" ‚Üí Click "Set" next to Application ID URI
  5. Step 5: Set URI to: api://[CLIENT_ID]/access_as_user ‚Üí Click Save
  6. Step 6: Click "Add a scope" ‚Üí Enter scope name: access_as_user ‚Üí Click "Add scope"
  7. Step 7: Create another app registration for your Angular app ‚Üí Select "Single Page Application" ‚Üí Add Redirect URI: http://localhost:4200/
  8. Step 8: Copy Angular App Client ID
  9. Step 9: In Angular app registration ‚Üí Click "API permissions" ‚Üí Click "Add a permission"
  10. Step 10: Select your Backend API ‚Üí Select "access_as_user" scope ‚Üí Click "Add permissions"
  11. Step 11: In .NET API, install NuGet package: Microsoft.Identity.Web
  12. Step 12: Configure in Program.cs: builder.AddMicrosoftIdentityWebApiAuthentication(configuration)
  13. Step 13: Add [Authorize] attribute to protected endpoints
  14. Step 14: In Angular, install MSAL: npm install @azure/msal-angular @azure/msal-browser
  15. Step 15: Configure MSAL with Tenant ID and Client ID ‚Üí Login users
  16. Step 16: When calling API, include token in Authorization header: Authorization: Bearer [TOKEN]
User ‚Üí Angular Login ‚Üí Azure AD validates ‚Üí JWT Token Issued
Angular ‚Üí API (with Token in header) ‚Üí API validates token ‚Üí Return secure data
Token Types:
  • ‚úì ID Token ‚Äì for user authentication (shows who user is)
  • ‚úì Access Token ‚Äì for calling secured APIs (proves user permission to access)

12. Monitoring & Logging ‚Äì Application Insights
‚ñº
What it is:

Azure's monitoring service that tracks everything happening in your application. It monitors API requests, errors, database queries, performance, and external dependencies. You can see what's happening in real-time and get alerts when problems occur.

Why use it:
  • ‚úì Track performance metrics (response times, throughput)
  • ‚úì Monitor errors and exceptions (see crashes immediately)
  • ‚úì Track database calls and API calls (see what's slow)
  • ‚úì Create alerts (get notified when problems happen)
  • ‚úì Monitor both frontend and backend (complete visibility)
  • ‚úì Dependency tracking (see which services call which)
Backend (.NET API):

Enable: Option 1: In App Service, enable Application Insights in settings. Option 2: Install NuGet package Microsoft.ApplicationInsights.AspNetCore, add builder.Services.AddApplicationInsightsTelemetry(); in Program.cs.

Tracks: API requests and response times, exceptions and errors, SQL database calls and performance, external API dependencies, custom logs via ILogger.

Frontend (Angular):

Enable: npm install @microsoft/applicationinsights-web, add connection string to your environment configuration, enable automatic route tracking.

Tracks: Page views and navigation, JavaScript errors, performance metrics, custom user events.

How to implement (Step by step) ‚ñº
  1. Step 1: Go to your App Service ‚Üí Click "Application Insights" in left menu
  2. Step 2: Click "Enable" ‚Üí Click "Apply" ‚Üí Wait for Azure to create Application Insights instance
  3. Step 3: Go to Application Insights resource ‚Üí Click "Logs" to view data
  4. Step 4: In Logs, run KQL queries: requests, exceptions, pageViews, dependencies
  5. Step 5: Create custom dashboards to monitor important metrics
  6. Step 6: Go to "Alerts" ‚Üí Click "Create alert rule" ‚Üí Set conditions (example: error rate > 5%)
  7. Step 7: Set action (email, SMS, webhook) to get notified of problems
  8. Step 8: For Angular, copy Application Insights connection string from Portal
  9. Step 9: In Angular environment.prod.ts, add instrumentationKey: '[YOUR_KEY]'
  10. Step 10: Install and configure MSAL in Angular to track authenticated user activities
View Logs:

Azure Portal ‚Üí Application Insights ‚Üí Logs. Query using KQL (Kusto Query Language):

  • ‚úì requests ‚Äì all API requests
  • ‚úì exceptions ‚Äì all errors and crashes
  • ‚úì pageViews ‚Äì user page navigations
  • ‚úì dependencies ‚Äì external API calls, SQL queries

13. CI/CD ‚Äì Azure DevOps Pipelines
‚ñº
What it is:

An automated system that builds your code, runs tests, creates Docker images, and deploys everything to Azure automatically when you push code to GitHub/Azure DevOps. No manual deployment needed ‚Äì push code ‚Üí automated deployment happens ‚Üí your app is live in seconds.

Why use it:
  • ‚úì Automated builds (code is compiled automatically)
  • ‚úì Faster deployments (deploy in seconds instead of manual hours)
  • ‚úì Consistent releases (same process every time)
  • ‚úì Separate environments (Dev/QA/Prod are isolated)
  • ‚úì Reduces manual errors (no human mistakes)
  • ‚úì Automatic testing (run unit tests before deploy)
Continuous Integration (CI):

Purpose: Automatically build, test, and validate code when changes are pushed.
Trigger: Code push to repository ‚Üí YAML pipeline automatically triggers
What happens: Pull latest code ‚Üí Restore NuGet packages ‚Üí Build project ‚Üí Run unit tests ‚Üí Create Docker image ‚Üí Push to Azure Container Registry

Continuous Deployment (CD):

Purpose: Deploy the built artifact to Azure environments automatically.
Process: Take build artifact from CI ‚Üí Deploy to App Service for APIs ‚Üí Deploy to Static Web Apps for Angular ‚Üí Run smoke tests to verify deployment

How to implement (Step by step) ‚ñº
  1. Step 1: Go to Azure DevOps ‚Üí Click "Pipelines" ‚Üí Click "Create Pipeline"
  2. Step 2: Select your GitHub/Azure DevOps repository
  3. Step 3: Select "ASP.NET Core" or "Node.js" template (based on your project)
  4. Step 4: Pipeline generates azure-pipelines.yml file with build steps
  5. Step 5: Add Docker steps to create Docker image (if using Docker)
  6. Step 6: Add steps to push Docker image to Azure Container Registry (ACR)
  7. Step 7: Click "Save and run" ‚Üí Pipeline runs and builds your code
  8. Step 8: After CI completes, create Release pipeline (or use Deployment jobs in YAML)
  9. Step 9: Configure deployment stages: Dev ‚Üí QA ‚Üí Production
  10. Step 10: Each stage deploys artifact to corresponding Azure environment
  11. Step 11: Add approval gates (manual approval before production deployment)
  12. Step 12: Configure secrets in pipeline (connection strings, API keys) using Azure Key Vault
Docker Deployment Flow
1. Developer pushes code to GitHub
2. CI Pipeline triggers automatically
3. Build Docker image from Dockerfile
4. Push Docker image to Azure Container Registry (ACR)
5. CD Pipeline triggers automatically
6. Update App Service with new image tag
7. App Service pulls image from ACR
8. App Service runs container
9. New version is live!
Developer Pushes Code ‚Üí GitHub ‚Üí CI Pipeline (Build) ‚Üí CD Pipeline (Deploy) ‚Üí Live on Azure

14. Workflow Automation ‚Äì Azure Logic Apps
‚ñº
What it is:

A no-code/low-code service to build automated workflows using a visual designer. No programming needed ‚Äì just drag and drop triggers and actions. Perfect for integration between systems and automating repetitive tasks.

Why use it:
  • ‚úì No-code automation (no programming skills needed)
  • ‚úì Visual designer (easy to understand)
  • ‚úì Built-in connectors (100+ pre-built integrations)
  • ‚úì Triggers and actions (when this happens, do that)
  • ‚úì Save time and money (automate manual tasks)
Common Triggers (What starts the workflow):
  • ‚úì When HTTP request arrives
  • ‚úì On schedule (Cron expression like "0 9 * * MON" = every Monday at 9 AM)
  • ‚úì When message arrives in Service Bus
  • ‚úì When file created in Blob Storage
  • ‚úì When email arrives in Outlook
  • ‚úì When item created in SharePoint
  • ‚úì When record created in SQL Database (using SQL trigger)
Common Actions (What the workflow does):
  • ‚úì Send Email (via Outlook or Gmail)
  • ‚úì Insert/Update database rows (via SQL)
  • ‚úì Send message to Service Bus or Queue
  • ‚úì Call external APIs (HTTP POST/GET)
  • ‚úì Create/Read files in Blob Storage
  • ‚úì Post message in Teams or Slack
  • ‚úì Conditions (IF/ELSE ‚Äì do different things based on conditions)
  • ‚úì Loops (For each ‚Äì repeat action for multiple items)
How to implement (Step by step) ‚ñº
  1. Step 1: Open Azure Portal ‚Üí Click "Create" ‚Üí Search "Logic App" ‚Üí Click Create
  2. Step 2: Fill in details: Resource Group, Logic App name (example: my-automation)
  3. Step 3: Click "Review + Create" ‚Üí Click "Create"
  4. Step 4: Open Logic App ‚Üí Click "Logic app designer"
  5. Step 5: Click "Start with a common trigger" ‚Üí Select trigger type (example: "When an email arrives")
  6. Step 6: Configure trigger (example: trigger on email from specific sender)
  7. Step 7: Click "Next step" ‚Üí Add action (example: "Send approval email")
  8. Step 8: Add more actions as needed (send email to manager, insert record in database, etc.)
  9. Step 9: Add conditions if needed (IF approver says yes, do X, else do Y)
  10. Step 10: Click "Save" ‚Üí Logic App is ready to run automatically
When to use Logic Apps:

System integration (connect multiple systems), notification workflows (send alerts), approval processes (get manager approval before proceeding), or any automation without writing code.

15. Infrastructure as Code ‚Äì Terraform
‚ñº
What it is:

Instead of clicking in Azure Portal manually, you write code files (.tf) that automatically create all your resources (App Services, databases, storage accounts, etc.). Code is version-controlled in Git, so you have history of all infrastructure changes.

Why use it:
  • ‚úì No manual resource creation (faster and less error-prone)
  • ‚úì Infrastructure defined in code (version controlled in Git)
  • ‚úì Version control history (see who changed what and when)
  • ‚úì Easy to recreate environments (code creates same environment reliably)
  • ‚úì Reduces human errors (code is consistent)
  • ‚úì Supports CI/CD automation (deploy infrastructure automatically)
  • ‚úì Easy team collaboration (everyone sees infrastructure code)
Key Concept - State File:

Terraform maintains a terraform.tfstate file that tracks what resources exist and their current configuration. This file is crucial ‚Äì never delete it! In production, store this file in Azure Storage Account (Remote Backend) so team members can access it.

Common Files:
  • ‚úì provider.tf ‚Äì Defines Azure provider and configuration
  • ‚úì main.tf ‚Äì Defines actual Azure resources (App Service, SQL Database, etc.)
  • ‚úì variables.tf ‚Äì Input variables (like function parameters)
  • ‚úì terraform.tfvars ‚Äì Actual values for variables per environment
  • ‚úì outputs.tf ‚Äì Output values after deployment (like function return values)
How to implement (Step by step) ‚ñº
  1. Step 1: Download and install Terraform from terraform.io
  2. Step 2: Create new folder for Terraform code (example: my-infrastructure)
  3. Step 3: Create provider.tf file with Azure provider configuration
  4. Step 4: Create main.tf file and define your resources (example: App Service, SQL Database)
  5. Step 5: Create variables.tf file with input variables
  6. Step 6: Create terraform.tfvars file with actual values for your environment
  7. Step 7: Create outputs.tf file to output important values (API URL, database name, etc.)
  8. Step 8: In terminal, run: terraform init (initializes Terraform)
  9. Step 9: Run: terraform plan (shows what resources will be created)
  10. Step 10: Run: terraform apply (actually creates resources in Azure)
  11. Step 11: Push code to Git ‚Üí Add pipeline to deploy infrastructure automatically
  12. Step 12: In future, to destroy all resources: terraform destroy
Implementation Steps
1. Install Terraform locally
2. Create provider configuration (Azure provider)
3. Define resources in main.tf
4. Define variables in variables.tf
5. Provide values in terraform.tfvars
6. Run terraform init, terraform plan, terraform apply
7. Push to Git ‚Üí CI/CD deploys automatically
Terraform Code ‚Üí Git Repository ‚Üí CI/CD Pipeline ‚Üí Azure API ‚Üí Infrastructure Created

Complete Architecture Overview
‚ñº
Main Request Flow (Read Operations):

Angular (Static Web App) ‚Üí APIM Gateway ‚Üí App Services (Microservices) ‚Üí Redis Cache ‚Üí Azure SQL Database

Cache Logic: Cache Hit (data in Redis) ‚Üí Return from Redis immediately (fast!) | Cache Miss (data not in Redis) ‚Üí Query from SQL ‚Üí Store in Redis ‚Üí Return to user

File & Background Processing Flow:

Angular ‚Üí API ‚Üí Blob Storage (file saved) ‚Üí Queue (message added) ‚Üí Azure Function (processes) ‚Üí Update Database

Monitoring Everything:

Application Insights monitors Angular UI (frontend), APIM (gateway), App Services (APIs), Azure Functions (background jobs), SQL Database (data queries), and all external dependencies. All data flows through monitoring for complete visibility.
