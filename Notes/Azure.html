What it is: Azure App Service is a fully managed PaaS used to host Web Apps and APIs without managing servers (IIS, OS, patching, load balancer).

Flow: Developer → Azure DevOps Pipeline → Azure App Service → User API

Why needed: No server management, auto scaling, secure configuration, zero-downtime deployments, easy CI/CD integration, production-ready monitoring.

App Service Plan Defines computing resources like CPU, RAM, scaling capability, and pricing tier;
One App Service Plan can host multiple App Services sharing the same resources. 		Choose based on traffic and production requirements
Deployment Slots Separate environments (Staging/Production) inside the same App Service for zero-downtime deployments. Deploy to staging, test safely, then swap to production. Supports slot-specific configuration (DB, API keys) to prevent environment mixing. Blue-Green deployments, safe production releases
Important Setting Description
Application Settings Store environment variables (ConnectionStrings, API Keys, ASPNETCORE_ENVIRONMENT). Overrides appsettings.json. Never store secrets in code.
Connection Strings Secure storage for database connections (Azure SQL, MySQL, etc.)
Scaling (Vertical & Horizontal) Vertical: Increase CPU/RAM by upgrading plan.
Horizontal: Increase instance count for load balancing.
Always On Keeps your application warm and continuously running to avoid cold start delays in production. Enable for APIs.
Deployment Slots Deploy to Staging → Test → Swap to Production without downtime.
Managed Identity Azure creates an identity for your app automatically to securely access Azure SQL / Key Vault without storing credentials.
App Service → Uses Managed Identity → Authenticates via Azure AD → Access Azure SQL / Key Vault
Application Insights Monitor logs, performance metrics, failed request tracking, exceptions, live traffic of our app
User → App Service → Application Insights collects telemetry → Azure Monitor stores data → You analyze in portal

  5. Azure DevOps → Create Build Pipeline → Use DotNetCoreCLI@2 (publish)
  6. Add AzureWebApp@1 task → Select Service Connection → Provide App Name
  7. Pipeline Flow: Code Push → Build → Publish → Deploy to App Service
  8. For Production: Deploy to Staging Slot → Validate → Swap with Production

SECTION 2: Cloud Database – Azure SQL Database ▼

What it is: It is SQL Server in the cloud, where you manage only the database and Azure handles the infrastructure and updates.

Flow: Application (App Service / API) → Connection String / Managed Identity → Azure SQL Logical Server → Azure SQL Database → Data Storage

Why needed: No infrastructure management, automatic backups, built-in high availability, strong security, easy scaling, cloud-ready database solution.

Component Description Use Cases
Logical Server Management container that holds multiple databases, firewall rules, and admin login. Not a physical server. Organize multiple databases under one server, manage access and security.
Database Actual storage of tables, indexes, stored procedures, views, and data (same as SQL Server). Application data storage for web apps, APIs, enterprise systems.
vCore / DTU Model Pricing and performance model. vCore allows selecting CPU, RAM, and storage separately. DTU is bundled model. Choose performance based on workload and production traffic.
Service Tier General Purpose, Business Critical, Hyperscale — different performance and availability levels. Standard apps (General), low-latency enterprise apps (Business Critical), very large DBs (Hyperscale).
Important Feature Description
High Availability Built-in replicas and automatic failover with 99.99% SLA. No manual clustering required.
Automatic Backups Full, differential, and log backups automatically. Supports point-in-time restore.
Scaling Increase vCores, storage, or change tier with minimal downtime.
Security Firewall rules, Azure AD authentication, encryption at rest (TDE), encryption in transit (TLS).
Managed Identity Support Allows App Service to connect using Azure AD without storing DB passwords.
Monitoring & Tuning Integrated with Azure Monitor, Query Performance Insight, automatic index recommendations.
Point-in-Time Restore Restore database to a specific time within retention period (7–35 days default).

How to implement:

  1. Azure Portal → Create → "Azure SQL" → Choose "Single Database"
  2. Create or select Logical Server → Set admin username/password
  3. Select pricing model (vCore recommended) and service tier
  4. Configure networking → Allow specific IP or Azure services
  5. Enable Azure AD authentication (recommended for enterprise)
  6. Connect from App Service using Connection String or Managed Identity
  7. Monitor performance using Query Performance Insight and Azure Monitor

SECTION 3: Serverless Computing – Azure Functions ▼

What it is: A cloud service that runs your code automatically when something triggers it. No server management, pay only when it runs.

Flow: Event occurs (HTTP / Timer / Queue / Blob / Cosmos DB) → Function triggers → code runs → output written to target (DB / Queue / Blob)

Why needed: No server management, pay only when code runs, automatically scales with load, handles background tasks without blocking main API, supports many triggers to react to events across Azure services.

Trigger Type When it fires Use case
HTTP Trigger HTTP request arrives Lightweight API endpoints
Timer Trigger On a schedule (Cron expression) Nightly reports, cleanup jobs
Queue Trigger New message in Queue Storage Simple background processing, image resize
Service Bus Trigger Message in Service Bus queue/topic subscription Enterprise messaging between microservices
Blob Trigger File uploaded or modified in Blob Storage Process images, scan documents
Cosmos DB Trigger Data inserted or updated (Change Feed) Sync data, send notifications on data change
Feature Description
HTTP Trigger Security Function Key acts as an API password. Pass via query string ?code=FUNCTION_KEY or header x-functions-key: KEY. Levels: Anonymous (no key), Function (function key), Admin (master key)
Input Bindings Automatically pull data into your function — e.g. read a Blob file or Cosmos DB document without writing connection code
Output Bindings Automatically push data out of your function — e.g. write to Queue, Blob, Cosmos DB, or Service Bus without writing connection code
Durable Functions Extension of Azure Functions for stateful workflows. Executes functions sequentially (A → B → C) or in parallel (fan-out/fan-in). Automatically manages state, retries, and checkpoints.
Example (Sequential): Order Processing → Check Inventory → Process Payment → Send Confirmation Email.
Example (Parallel): Resize 10 images simultaneously, then generate final report.
Retry Policy Automatically retries on failure — configure max retry count and delay. On Queue/Service Bus trigger, failed messages move to Dead Letter Queue after max retries

How to implement:

  1. Azure Portal → Create Storage Account (required for runtime logs & scaling)
  2. Azure Portal → Create Function App → Select .NET Runtime → Choose Hosting Plan (Consumption / Premium / App Service) → Attach Storage Account
  3. Function App → Configuration → Add App Settings / Connection Strings (Recommended: store secrets in Key Vault)
  4. Visual Studio → Create Azure Functions Project → Select Trigger (HTTP / Timer / Service Bus etc.)
  5. Write function using [FunctionName] + Trigger attribute (e.g. [HttpTrigger], [ServiceBusTrigger])
  6. Store local values in local.settings.json → Test locally
  7. Right-click Project → Publish → Select Existing Azure Function App
  8. Monitor execution via Application Insights / Monitor tab

SECTION 4: Enterprise Messaging – Azure Service Bus ▼

What it is: Azure Service Bus helps applications send messages to each other safely using queues and topics for background processing.

Flow: Microservice 1 (Publisher) → Azure Service Bus Queue/Topic (Message Storage) → Microservice 2 & 3 (Subscribers)

Why needed: Decouples microservices so they don't depend on each other directly, guarantees message delivery even if receiver is down, handles retries automatically, prevents data loss, and allows multiple services to react to the same event independently.

Type Description Use Cases
Queue (1-to-1) One sender, one receiver — message consumed once by a single consumer Order processing, payment processing, background jobs, email sending, report generation, image processing, inventory updates
Topic (1-to-many) One sender, multiple subscriptions — each subscription gets its own independent copy Notifications (SMS/Email), microservices communication, event broadcasting, audit logging, analytics processing, sending data to multiple systems
Feature Description
Peek-Lock Message locked during processing. If processing fails or lock expires, it becomes available again and is retried. In a Topic, each subscription has its own copy — if one fails and retries, others are not affected.
Automatic Retries Retries on failure before moving to Dead Letter Queue
Dead Letter Queue After exceeding max delivery count, message moves here automatically — inspect and retry manually
Duplicate Detection Azure service bus uses MessageId to detect and discard duplicate messages sent by publisher
Sessions Messages with the same SessionId are locked to one session and processed one by one in the exact order they were sent
Subscription Filters Routes messages to specific subscriptions based on message properties e.g. channel = 'sms'. If no filter is set, all subscriptions receive all messages — broadcast model
Scheduled Messages Deliver a message at a future time — set ScheduledEnqueueTime on the message
Orchestrator Pattern To execute steps in order (e.g., SMS → Email → Letter), use a single queue with one consumer so steps run sequentially. Complete the message only after all steps succeed. Since Service Bus provides at-least-once delivery, implement idempotency to avoid duplicates — store each completed step using CorrelationId + StepName in a database and skip already completed steps during retries. You must manage state manually; alternatively, use a Durable Function.

How to implement:

  1. Azure Portal → Create → "Service Bus" → fill in Resource Group, Namespace name, pricing tier (Standard for Topics, Basic for Queues only)
  2. Queue: Namespace → Queues → Add Queue → set name and Max Delivery Count
  3. Topic: Namespace → Topics → Add Topic → open Topic → Subscriptions → Add Subscription per consumer (e.g. sms-sub, email-sub)
  4. Filters: open Subscription → Filters → Add Filter → e.g. channel = 'sms'
  5. Sessions: enable "Requires Session" on Queue/Topic at creation
  6. Duplicate Detection: enable "Requires Duplicate Detection" on Queue/Topic at creation
  7. Namespace → Shared Access Policies → copy Primary Connection String → store in Key Vault
  8. .NET API → install NuGet: Azure.Messaging.ServiceBus → register in Program.cs: builder.Services.AddSingleton(new ServiceBusClient(connectionString))
  9. Send: inject ServiceBusClient → create sender → call SendMessageAsync(new ServiceBusMessage(body))
  10. Receive: create Azure Function with "Service Bus Trigger" → binds to your queue or topic subscription → processes automatically

SECTION 6: Storage Account – Blob & Queue Storage ▼

What it is: A cloud storage service that holds your data. Inside it you can create Blob Storage (for files) and Queue Storage (for messages). Don't store files in your database — use Blob Storage, it's cheaper and faster. Each file gets a unique URL.

Blob Flow: Angular → API → Blob Storage → save file URL in SQL DB

Queue Flow: User → API → Queue (message added) → Azure Function (processes) → Database Updated

Type What it stores Use case
Blob Storage Files — images, videos, PDFs, backups User uploads, static files, logs
Queue Storage Short text messages (max 64 KB) Pass tasks between services, background jobs
Feature Description
Blob Access Tiers Hot — accessed frequently | Cool — accessed occasionally | Archive — rarely accessed, cheapest but slow to retrieve
Container Access Private (no public access), Blob (anyone can read a file by URL), Container (anyone can list and read all files)
SAS Token Temporary URL with expiry and permissions — share a file securely without making the container public
Lifecycle Management Auto-move blobs between tiers or delete after X days — e.g. Cool after 30 days → Archive after 90 → Delete after 365
Blob Trigger Runs a azure function automatically when a file is uploaded or modified
Queue Trigger Runs a azure function automatically when a new message arrives in the queue. Auto-retries on failure (default 5 times) → then moves to Poison Queue for manual inspection.
Visibility Timeout (Queue) Message becomes hidden while being processed. If not deleted in time, it reappears for another consumer to retry.

How to implement:

  1. Azure Portal → Create Storage Account → Choose Region → Select Redundancy (LRS / GRS)
  2. Blob: Storage Account → Containers → Create Container → Set access level
  3. Queue: Storage Account → Queues → Create Queue → Give it a name
  4. Storage Account → Access Keys → Copy Connection String → store in Key Vault / local.settings.json
  5. Install NuGet: Azure.Storage.Blobs or Azure.Storage.Queues
  6. Use BlobServiceClient to upload/download files or QueueClient to send/receive messages
  7. For Azure Functions: use [BlobTrigger] or [QueueTrigger] — Azure handles reading automatically

Queue benefits:

  • ✓ Handles background tasks like order processing, email sending, and file processing without blocking the main API.
  • ✓ Improves system performance by responding instantly to users while heavy work runs asynchronously.
  • ✓ Increases reliability with built-in retry and poison queue support if processing fails.
  • ✓ Enables scalable microservices where multiple workers can process messages independently.

Download Flow (Blob): Angular → API generates SAS URL → Angular downloads directly from Blob

SECTION 8: Identity & Access – Azure AD (Microsoft Entra ID) ▼

What it is: It is service that manages users, login (Auth and Authorization), MFA, and issues secure tokens for accessing apps/APIs.

Flow: User → Angular (MSAL) → Azure AD Login (+MFA) → Access Token (JWT)→ Angular sends JWT → .NET Web API (Validate JWT) → Response

Why needed: centralized login, secure token-based API access, enterprise MFA + RBAC, no passwords stored in your app

Token/Concept Description
JWT Token format (Header.Payload.Signature). In Azure AD, both ID Token and Access Token are JWT strings.
ID Token Used by Angular to identify user (name/email). Not meant for calling APIs. Authentication
Access Token Used to call Web API. Sent as Authorization: Bearer <token>. Web API validates this JWT. Authorization
Authorization Code Temporary one-time code returned after login; Angular (MSAL) exchanges it for tokens securely (PKCE). Never handle it manually.
PKCE Security step for SPA: prevents code interception; MSAL manages code_verifier and code_challenge.
RBAC (Role Based Access Control) Flow: Admin assigns role in Azure AD → role added into JWT (roles claim) → Web API checks role using [Authorize(Roles="Admin")]. RBAC controls what user can do after login.
MFA (Multi-Factor Authentication) MFA happens during Azure AD login (before tokens are issued). If policy requires MFA → user must complete OTP/Authenticator → only then Azure AD issues tokens. MFA controls how strongly user is verified.

How to implement:

  1. Register Web API App (Backend Identity):
    Entra ID → App registrations → New registration → Name your API → Register → Copy Tenant ID and API Client ID
  2. Expose API Scope (App scope - Which client apps can access our API?):
    API App → Expose an API → Set Application ID URI → Add scope access_as_user
    Scope format becomes: api://<API_CLIENT_ID>/access_as_user
  3. Create Roles (RBAC – What login user can do?):
    API App → App roles → Create roles like Admin, User
    Enterprise Applications → Open your API → Users & Groups → Assign users/groups to roles
  4. Register Angular SPA App (Frontend Identity):
    App registrations → New registration → Register Angular app → Copy Angular Client ID
  5. Configure Redirect URI:
    Angular App → Authentication → Add platform → Single Page Application → Add http://localhost:4200
  6. Grant Angular Permission to Call API:
    Angular App → API permissions → Add permission → My APIs → Select your API → Choose access_as_user → (Optional) Click Grant Admin Consent
  7. Configure Angular (MSAL Setup):
    Install MSAL packages → Configure:
    • clientId = Angular Client ID
    • authority = https://login.microsoftonline.com/<TenantID>
    • redirectUri = http://localhost:4200
    • scope = api://<API_CLIENT_ID>/access_as_user
    Use MSAL interceptor to automatically attach Authorization: Bearer <access_token> in API calls.
  8. Configure .NET Web API (JWT Validation):
    Add JWT Bearer Authentication with:
    • Authority = https://login.microsoftonline.com/<TenantID>
    • Audience = API Client ID
    Protect endpoints using: [Authorize]
    Protect role-based endpoints using: [Authorize(Roles = "Admin")]
  9. Enable MFA (Extra Security Layer):
    Entra ID → Security → Conditional Access → Create Policy → Select Users → Select Cloud App (your app) → Require Multi-Factor Authentication
  10. Final Runtime Flow:
    User logs in (Password + MFA) → Azure AD issues JWT → JWT contains scp (scope) + roles → Angular calls API with Access Token → API validates token + checks roles → Access granted or denied.

SECTION 12: Caching – Azure Redis Cache ▼

What it is: Instead of querying the database every time, store frequently accessed data in Redis (RAM memory) which is 1000x faster. Perfect for data that doesn't change often.

Flow: API receives request → Check Redis first → If found (Cache Hit): return data instantly → If not found (Cache Miss): query Database → store result in Redis with TTL → return data to user → On data update/delete: remove cache key so next request fetches fresh data

Why needed: Database queries are slow and expensive — Redis stores results in RAM which is 1000x faster, reduces database load, improves API response time, and keeps sessions working correctly across multiple servers behind a load balancer.

Features Description
Reduce Database Load Stores frequently used data in memory so the same queries don’t hit the database again and again.
High Performance Very fast API responses because data is read from RAM (in-memory) instead of disk/database.
Distributed Caching All servers connect to the same Redis, so cached data is shared across multiple servers behind a load balancer.
Session Storage Keeps user session/login data centrally, so sessions work properly even when requests hit different servers.
TTL (Expiry Support) Supports automatic expiry (TTL) so cached data is removed after a set time to avoid stale data.
Advanced Enterprise Capabilities Supports Pub/Sub (real-time messaging), distributed locking (avoid duplicate processing), scalability, high availability (replication + failover), and cost optimization by reducing database/infrastructure load.

Cache Logic:

  • Cache Hit: Data found in Redis → return immediately (fast!)
  • Cache Miss: Not in Redis → query database → store in Redis with TTL → return to user
  • Cache Invalidation: On update/delete → clear cache so next query fetches fresh data
  • TTL (Time To Live): How long data stays in cache (e.g. 1 hour). Balances performance vs freshness.

How to implement:

  1. Azure Portal → Create → "Azure Cache for Redis" → fill in Resource Group, Cache name, pricing tier
  2. Redis Cache → Settings → Access Keys → copy Primary Connection String
  3. .NET API → install NuGet: StackExchange.Redis
  4. Check Redis first → if hit, return; if miss, query DB → store in Redis with TTL → return
  5. On data change → invalidate (delete) the relevant cache key

SECTION 13: Frontend Hosting – Azure Static Web Apps ▼

What it is: Managed service to host frontend like Angular, React, or Vue apps. Provides automatic HTTPS, global CDN, and auto-deploy on GitHub push.

Flow: Developer pushes code to GitHub → Pipeline builds app → Static Web App deploys to CDN → User opens browser → Content served from nearest CDN location → API calls go to Backend

Why needed: No server to manage, free HTTPS, globally fast loading via CDN, auto-deploys on every push, scales automatically with zero configuration.

Important Setting Description
Custom Domains Map your own domain (e.g. myapp.com) to your Static Web App with free SSL certificate automatically applied.
Preview Environments Every Pull Request automatically gets its own temporary URL for testing before merging to production.
Authentication Built-in login support with Azure AD, GitHub, Google — no backend code needed. Users can be restricted by role.
Linked API (Azure Functions) Attach an Azure Functions project as your backend API — both frontend and API deploy together under one service.
Environment Variables Set app settings per environment (dev/staging/prod) — injected at build time, never stored in code.
Routing Rules Define custom routes in staticwebapp.config.json — handle 404 fallback, redirects, and header rules.

How to implement:

  1. Create Static Web App in Azure Portal
  2. Copy Deployment Token
  3. Add token as secret variable in Azure DevOps Pipeline
  4. Add azure-pipelines.yml with $(SWA_DEPLOYMENT_TOKEN) app_location and output_location
  5. Push code → Pipeline builds and deploys automatically
  6. Update environment.prod.ts with API base URL
  7. Enable CORS in backend

SECTION 14: NoSQL Database – Azure Cosmos DB ▼

What it is: A globally distributed NoSQL database that stores data as JSON documents instead of tables and rows. Use it when your data has no fixed structure or needs to scale massively.

Flow: Angular → API → CosmosClient → Container → JSON Documents stored/retrieved

Why needed: No fixed schema so data structure can change freely, scales globally with one click, reads are extremely fast, and handles massive amounts of data that relational databases cannot handle efficiently.

Concept Description SQL Equivalent
Database Top-level container that holds multiple containers. SQL Server instance
Container Holds JSON documents. No fixed schema — each document can have different fields. Table
Item / Document A single JSON record inside a container. Each must have a unique id. Row
Partition Key Field used to split data across partitions for performance. Always include it in queries. Choose high variety values like userId — never low variety like status. No direct equivalent
Throughput (RU/s) Request Units per second — controls how many reads/writes per second. Set per container. Can be manual or autoscale. No direct equivalent
Important Feature Description
Global Distribution Replicate data to any Azure region with one click. Reads are served from the nearest region automatically.
TTL (Time to Live) Auto-delete documents after X seconds. Useful for sessions, logs, and temporary data.
Change Feed Ordered stream of all inserts/updates. Use with Azure Functions to react to data changes automatically.
Consistency Levels Controls how up-to-date reads are. Default is Session — you always read your own writes.

SQL vs Cosmos DB — When to use which:

  • Azure SQL: Structured data, complex joins, strict schema — banking, HR, ERP systems
  • Cosmos DB: Flexible schema, massive scale, global users, high-speed reads — e-commerce, IoT, real-time apps, user sessions

How to implement:

  1. Azure Portal → Create → "Azure Cosmos DB" → select API (NoSQL for JSON)
  2. Set Resource Group, Account name, Region → select Capacity mode (Provisioned or Serverless)
  3. Data Explorer → New Database → New Container → set name and Partition key (e.g. /userId)
  4. Install NuGet: Microsoft.Azure.Cosmos
  5. Register in Program.cs: builder.Services.AddSingleton<CosmosClient>(new CosmosClient(connectionString))
  6. Inject CosmosClient → get container → call CreateItemAsync, ReadItemAsync, GetItemQueryIterator
  7. Store connection string in Azure Key Vault — never hardcode it

Common Operations (.NET SDK):

  • Create: container.CreateItemAsync(item, new PartitionKey(item.UserId))
  • Read: container.ReadItemAsync<T>(id, new PartitionKey(userId))
  • Query: container.GetItemQueryIterator<T>(new QueryDefinition("SELECT * FROM c WHERE c.userId = @id"))
  • Update: container.UpsertItemAsync(item, new PartitionKey(item.UserId))
  • Delete: container.DeleteItemAsync<T>(id, new PartitionKey(userId))

SECTION 15: Infrastructure as Code – Terraform ▼

What it is: Terraform allows you to create & manage Azure resources using .tf configuration files instead of manually setting them up in Azure Portal. Version-controlled in Git with full history changes.

Flow: Terraform code → Git → CI/CD Pipeline → Azure API → Infrastructure created automatically on Azure portal.

Why needed: Avoids manual setup in Azure Portal, infrastructure is version-controlled in Git so changes are tracked, same setup can be redeployed to any environment (dev/staging/prod) without mistakes, and the whole team works from the same configuration.

File Purpose
provider.tf Defines Azure provider and configuration
main.tf Defines actual Azure resources (App Service, SQL, Storage, etc.)
variables.tf Input variables (like function parameters)
terraform.tfvars Actual values for variables per environment
outputs.tf Output values after deploy (API URL, DB name, etc.)
terraform.tfstate Tracks what resources exist — never delete! Store in Azure Storage for team access (Remote Backend)

Key Commands:

  • terraform init — initialize Terraform
  • terraform plan — preview what will be created/changed
  • terraform apply — create/update resources in Azure
  • terraform destroy — delete all resources

How to implement:

  1. Install Terraform from terraform.io
  2. Create provider.tf, main.tf, variables.tf, terraform.tfvars, outputs.tf
  3. Run terraform init → terraform plan → terraform apply
  4. Push code to Git → add CI/CD pipeline to deploy infrastructure automatically
